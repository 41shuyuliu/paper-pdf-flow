# 网页文献输入源扩展设计（OA / DOI / 当前标签页）

## 1. 版本
- 文档版本：`v0.1`
- 日期：`2026-03-24`
- 适用范围：`paper_pdf_flow_skill`
- 目标读者：产品设计、插件开发、后端开发、抽取链路维护者

## 2. 背景

当前项目已经完成以下能力：
- 以单篇 `PDF` 为输入；
- 抽取正文、图注与实验信号；
- 生成固定结构的中文“论文解读”Markdown；
- 插件与后端共享同一套正式产物契约。

当前痛点在于：
- 用户必须先拿到 PDF，增加了使用门槛；
- 对开放获取（OA）论文来说，网页 HTML 往往比 PDF 更结构化、更适合抽取；
- 用户已经打开合法可访问的论文网页时，重复下载 PDF 再上传属于冗余操作；
- 一些论文网页拥有更完整的元信息、章节结构、图注和补充说明，适合直接读取。

因此，需要把系统从“PDF-only 输入”升级为“多输入源系统”，其中网页文献应成为一级输入源，而 PDF 降级为兜底输入源之一。

## 3. 目标

### 3.1 总目标
- 在不改变最终产物契约的前提下，将输入源从单一 `PDF` 扩展为：
  - `当前标签页网页`
  - `网页 URL`
  - `DOI`
  - `开放 PDF（兜底）`

### 3.2 产品目标
- 让用户在“已打开论文网页”的场景下，可直接一键生成论文解读；
- 让用户在“只有 DOI / URL”的场景下，无需手动寻找 PDF；
- 对开放获取网页优先使用 HTML 正文，而不是强制退回 PDF；
- 保持最终输出仍为单文件 Markdown 的“论文解读”正式稿。

### 3.3 技术目标
- 统一不同输入源的内部数据结构；
- 复用现有 `core/` 中的 prompt 组装和最终生成逻辑；
- 将“网页抽取”设计为可扩展适配器体系，而不是一次性写死在插件里；
- 让插件与后端都能逐步复用网页抽取能力。

## 4. 合规边界

本设计仅覆盖以下合法输入源：
- 开放获取（OA）网页全文；
- 用户当前已合法访问的网页内容（例如机构订阅、个人订阅、校园网访问、已登录授权页面）；
- DOI 可解析到的合法落地页、开放版本页面或开放 PDF。

本设计明确不包含以下内容：
- 任何绕过版权、登录、付费墙或访问控制的能力；
- 任何第三方盗版镜像站接入；
- 任何自动规避站点风控、验证码、身份校验的能力。

结论：
- “当前标签页读取”是允许且高价值的；
- “OA 网页读取”是允许且应优先支持的；
- “非法绕过访问控制的来源”不在本项目范围内。

## 5. 设计原则

- **正式产物唯一**：全项目只保留一种正式产物，即结构化“论文解读”Markdown。
- **输入源多样，内部结构统一**：不论来自 PDF、网页还是 DOI，最终都归一为统一的文档结构。
- **网页优先，PDF 兜底**：如网页抽取质量足够，应优先使用网页；仅在必要时回退到开放 PDF。
- **插件优先满足交互便捷性**：当前标签页读取应优先在插件中落地。
- **后端优先满足可批处理性**：URL / DOI 解析和服务端抓取应优先沉淀到后端。
- **适配器 + 通用抽取并存**：先用通用规则覆盖广度，再为高频站点补适配器提升精度。

## 6. 用户场景

### 6.1 场景 A：用户已打开 OA 论文网页
- 输入方式：点击插件 -> `从当前页面生成`
- 期望：无需下载 PDF，直接生成论文解读稿

### 6.2 场景 B：用户手里只有文献链接
- 输入方式：粘贴 URL -> 生成
- 期望：系统自动读取网页正文，并生成论文解读稿

### 6.3 场景 C：用户手里只有 DOI
- 输入方式：粘贴 DOI -> 生成
- 期望：系统自动解析 DOI -> 找到合法落地页或开放版本 -> 生成论文解读稿

### 6.4 场景 D：网页质量差或正文不完整
- 输入方式：URL / DOI
- 期望：系统自动回退到开放 PDF 或提示用户切换为 PDF 上传

### 6.5 场景 E：用户已合法登录出版社网页
- 输入方式：当前标签页
- 期望：插件直接读取用户当前可见正文，不要求系统代替用户登录

## 7. 范围定义

### 7.1 本期纳入范围
- 插件：
  - 从当前标签页提取正文与元信息；
  - 从 URL 读取网页；
- 后端：
  - 接收 URL / DOI 任务；
  - 获取网页正文或合法开放 PDF；
- `core/`：
  - 新增网页文献的归一化数据结构；
  - 新增网页抽取后的 prompt payload 构建；
- 文档：
  - 定义新的输入源类型、降级策略和验收口径。

### 7.2 明确不在本期范围
- 批量 DOI 批处理；
- 补充材料（Supplementary）深度解析；
- 表格高保真还原；
- 图片 OCR；
- 登录态自动维护；
- 任何非合规来源接入。

## 8. 总体架构

建议将系统拆分为五层：

### 8.1 `Source Resolver`
负责识别输入源类型：
- `pdf`
- `current_tab_html`
- `url_html`
- `doi`

职责：
- 判定输入类型；
- 将 `doi` 解析为落地页；
- 决定优先走网页还是 PDF；
- 返回一个规范化的“候选资源列表”。

### 8.2 `Fetcher`
负责获取原始内容：
- 插件内：读取当前页面 DOM；
- 后端：请求 URL，下载 HTML 或开放 PDF；
- 当输入为 DOI 时，先跟随 DOI 解析后的落地页。

职责：
- 获取 HTML 文本；
- 获取 meta / JSON-LD / structured data；
- 检测是否存在合法开放 PDF 链接；
- 返回原始抓取结果。

### 8.3 `Extractor`
负责从原始内容中抽取结构化论文信息。

内部拆成两类：
- **通用抽取器**：适用于大多数站点；
- **站点适配器**：适用于高频站点的定制规则。

### 8.4 `Normalizer`
负责将 PDF、网页、URL、DOI 最终统一成同一份内部数据结构。

### 8.5 `Interpreter`
复用现有“论文解读”生成逻辑：
- prompt 组装；
- 模型调用；
- 输出固定 Markdown。

## 9. 统一内部数据结构

建议引入统一的 `ArticleDocument` 概念，字段如下：

```json
{
  "source_type": "current_tab_html | url_html | doi | pdf",
  "source_url": "string",
  "canonical_url": "string",
  "doi": "string",
  "title": "string",
  "journal": "string",
  "authors": ["string"],
  "year": "string",
  "abstract": "string",
  "sections": [
    {
      "heading": "string",
      "level": 2,
      "text": "string"
    }
  ],
  "figures": [
    {
      "label": "Fig.1",
      "caption": "string"
    }
  ],
  "tables": [
    {
      "label": "Table 1",
      "caption": "string"
    }
  ],
  "full_text": "string",
  "signals": {
    "timepoints": ["string"],
    "doses": ["string"],
    "metrics": ["string"]
  },
  "quality": {
    "body_chars": 0,
    "section_count": 0,
    "figure_count": 0,
    "extractor": "generic | adapter:<site>",
    "fallback_used": false
  }
}
```

说明：
- `sections` 是网页输入的关键新增字段；
- `full_text` 继续保留，便于兼容现有 prompt 逻辑；
- `quality` 用于后续回退决策和调试。

## 10. 网页抽取策略

### 10.1 元信息抽取
优先级建议如下：
1. `meta` 标签（如 title、doi、citation_*）
2. `JSON-LD` / structured data
3. 页面主标题与副标题
4. 正文首屏启发式抽取

目标字段：
- 标题
- DOI
- 期刊
- 作者
- 年份

### 10.2 正文抽取
优先抽取：
- `article`
- `main`
- 明确的正文容器
- 章节标题与正文段落层级

需要识别的章节包括：
- Abstract
- Introduction
- Methods / Materials and Methods
- Results
- Discussion
- Conclusion

### 10.3 图注抽取
优先抽取：
- `figure` / `figcaption`
- 页面中明确的图标签与图注容器
- 高置信度“Figure / Fig.”标题块

### 10.4 表格抽取
本期只做轻量支持：
- 抽表题与简要说明；
- 不追求复杂表格的高保真结构恢复。

### 10.5 信号抽取
复用现有启发式规则，继续抽取：
- 时间点
- 剂量/浓度
- 指标关键词
- 数据集/实验术语

### 10.6 文本质量判定
建议设置基础阈值：
- 正文字符数过短 -> 视为低质量；
- 缺少 Methods / Results -> 降低置信度；
- 图注与章节数过少 -> 考虑回退；

## 11. 适配器体系

### 11.1 适配器目标
- 对高频 OA 站点提升抽取稳定性；
- 减少通用规则对 DOM 变化的敏感性；
- 保证章节与图注结构更完整。

### 11.2 首批建议支持站点
- `PMC`
- `arXiv`
- `bioRxiv / medRxiv`
- `PLOS`
- `Frontiers`
- `MDPI`

### 11.3 适配器选择顺序
1. 站点适配器命中 -> 走站点规则；
2. 若未命中 -> 走通用抽取器；
3. 若网页质量差 -> 尝试开放 PDF 回退；
4. 若仍失败 -> 返回用户可读错误并建议上传 PDF。

## 12. 回退策略

### 12.1 首选路径
- 当前标签页 / URL / DOI -> HTML 正文抽取 -> 生成论文解读稿

### 12.2 第一层回退
- HTML 正文不完整，但可发现合法开放 PDF -> 退到 PDF 抽取

### 12.3 第二层回退
- HTML 与 PDF 都无法得到足够正文 -> 返回失败

### 12.4 失败提示原则
- 必须告诉用户失败在哪一层：
  - DOI 解析失败
  - 网页获取失败
  - 网页正文抽取不足
  - 未发现合法开放 PDF

## 13. 与现有代码的对齐方式

### 13.1 `core/`
新增能力：
- `html` / `url` / `doi` 的文档归一化结构；
- 从 `ArticleDocument` 构造与现有 prompt 兼容的 payload；
- 保留现有 `paper_interpretation.py` 作为统一解释层。

### 13.2 `extension/`
新增入口：
- `从当前页面生成`
- `从 URL 生成`

插件职责：
- 读取当前页面 DOM；
- 在需要时发送 URL 给后端；
- 保留现有 PDF 上传入口，作为兼容模式。

### 13.3 `backend/`
建议扩展任务协议，支持以下字段：
- `source_type`
- `url`
- `doi`
- `html_snapshot`（可选，仅插件当前页模式）

后端职责：
- 处理 DOI / URL；
- 在服务端执行网页抓取与 HTML 抽取；
- 调用现有生成链路并返回最终稿。

## 14. 产品交互方案

### 14.1 插件主界面建议
当前 PDF 入口保留，同时增加三个来源按钮：
- `上传 PDF`
- `从当前页面生成`
- `从 URL 生成`

后续可在第二阶段补：
- `从 DOI 生成`

### 14.2 当前页面模式
用户流程：
1. 打开论文网页；
2. 点击插件；
3. 选择 `从当前页面生成`；
4. 插件提取 DOM / meta / 图注；
5. 直接调用模型或发送到后端；
6. 下载论文解读稿。

### 14.3 URL / DOI 模式
用户流程：
1. 粘贴 URL 或 DOI；
2. 系统解析落地页；
3. 读取网页正文；
4. 必要时回退开放 PDF；
5. 生成论文解读稿。

## 15. API 演进建议

建议在后端接口上新增对网页任务的兼容支持，而不是新起一套完全独立协议。

### 15.1 建议扩展 `POST /tasks`
在现有字段基础上增加：
- `source_type`：`pdf | url | doi | html_snapshot`
- `url`：网页地址
- `doi`：DOI 文本
- `html_snapshot`：插件当前页序列化 HTML（可选）

约束：
- `source_type=pdf` 时必须提供 `pdf`
- `source_type=url` 时必须提供 `url`
- `source_type=doi` 时必须提供 `doi`
- `source_type=html_snapshot` 时必须提供 `url` 或 `canonical_url` 以及 `html_snapshot`

### 15.2 兼容策略
- 保持原 PDF 任务继续可用；
- 新字段为增量扩展，不打断现有调用方。

## 16. 分阶段实施计划

### 阶段 1：插件当前页读取（推荐优先）
目标：
- 最快验证“无需 PDF”的价值；
- 优先支持合法已打开页面。

交付：
- 插件 UI 新增“从当前页面生成”；
- 新增通用 HTML 抽取；
- 输出仍为论文解读稿。

### 阶段 2：URL 读取
目标：
- 支持用户直接粘贴网页链接；
- 优先覆盖 OA 站点。

交付：
- 后端 / 插件支持 URL 输入；
- 增加网页质量判定与回退逻辑。

### 阶段 3：DOI 解析
目标：
- 让 DOI 成为一等输入。

交付：
- DOI -> 落地页解析；
- DOI -> OA 网页 / 开放 PDF 回退逻辑。

### 阶段 4：站点适配器增强
目标：
- 提高常见 OA 平台的正文、图注、章节抽取精度。

交付：
- 3~5 个高频站点适配器；
- 抽取质量指标和诊断日志。

## 17. 风险与应对

### 17.1 风险：网页结构高度异构
- 应对：通用规则 + 站点适配器双层架构。

### 17.2 风险：页面由前端脚本动态渲染
- 应对：插件优先读取当前已渲染 DOM；后端仅对静态可抓取页面做支持。

### 17.3 风险：图注和正文分离
- 应对：将图注抽取视为增强能力，不作为成功唯一条件。

### 17.4 风险：DOI 落地页不是全文页
- 应对：将 DOI 解析视为“资源发现”过程，而非直接保证全文获取。

### 17.5 风险：网页抽取质量不稳定
- 应对：增加质量评分与开放 PDF 回退。

### 17.6 风险：合规边界模糊
- 应对：文档与代码中都明确禁止任何绕过访问控制的实现。

## 18. 验收标准

### 18.1 阶段 1 验收
- 用户可在已打开论文网页时直接生成论文解读稿；
- 输出结构与当前 PDF 模式一致；
- 对正文充足的 OA 页面，成功率达到可演示水平；
- 页面读取失败时能给出清晰提示。

### 18.2 阶段 2 验收
- 粘贴 URL 可生成论文解读稿；
- 网页正文不足时能自动回退开放 PDF 或返回可读错误。

### 18.3 阶段 3 验收
- DOI 输入可解析到合法资源；
- 能区分“未找到开放资源”和“抽取失败”两类错误。

## 19. 推荐结论

推荐执行顺序为：
1. **先做“当前标签页读取”**；
2. 再做 `URL` 输入；
3. 最后补 `DOI` 解析。

原因：
- 当前标签页模式最贴近用户行为；
- 技术难度最低，能最快验证“跳过 PDF”是否真正提升体验；
- 这一路径天然支持“用户已合法访问”的网页，不需要系统承担登录和授权问题；
- 做完当前页模式后，抽取与归一化逻辑可继续复用到 URL / DOI 模式。
